# Requirements Document

## Introduction

The Recursive Memory (Agentic RAG) feature provides long-term memory for the Sesari agent by indexing past suggestions, their outcomes, and business context. This enables the agent to learn from previous actions, avoid repeating failed strategies, and provide increasingly relevant recommendations over time.

## Glossary

- **Memory_System**: The recursive memory component that stores and retrieves historical agent actions and business context
- **Growth_Play**: An actionable recommendation generated by the agent
- **Outcome_Record**: A document capturing whether a Growth_Play succeeded or failed
- **Strategy_Document**: Business DNA files including ICP, growth playbook, and brand voice
- **Performance_Summary**: Weekly aggregated business metrics and trends
- **Action_History**: Historical record of all Growth_Plays and their outcomes
- **Technical_Map**: Metadata defining signal schemas and integration configurations
- **RAG_Index**: Vector-based search index powered by Amazon Bedrock Knowledge Bases
- **S3_Bucket**: Amazon S3 storage containing all memory documents
- **Bedrock_KB**: Amazon Bedrock Knowledge Base service for vector storage and retrieval

## Requirements

### Requirement 1: Store Strategy Documents

**User Story:** As a SaaS founder, I want the agent to understand my business DNA, so that recommendations align with my ICP and growth playbook.

#### Acceptance Criteria

1. WHEN a Strategy_Document is uploaded, THE Memory_System SHALL store it in the S3_Bucket
2. THE Memory_System SHALL support ICP documents, growth playbooks, and brand voice guidelines
3. WHEN a Strategy_Document is updated, THE Memory_System SHALL replace the previous version
4. THE Memory_System SHALL index Strategy_Documents in the RAG_Index within 5 minutes of upload

### Requirement 2: Generate Performance Summaries

**User Story:** As a growth lead, I want weekly business summaries indexed, so that the agent can spot trends without processing raw transaction data.

#### Acceptance Criteria

1. WHEN Sunday arrives, THE Memory_System SHALL generate a Performance_Summary for the previous week
2. THE Performance_Summary SHALL include revenue change percentage, usage metrics by segment, and churn indicators
3. THE Memory_System SHALL store the Performance_Summary in the S3_Bucket
4. THE Memory_System SHALL index the Performance_Summary in the RAG_Index

### Requirement 3: Record Growth Play Actions

**User Story:** As a user, I want the agent to remember what it has tried before, so that it doesn't repeat the same failed strategies.

#### Acceptance Criteria

1. WHEN the agent generates a Growth_Play, THE Memory_System SHALL create an Action_History record
2. THE Action_History record SHALL include the Growth_Play description, timestamp, and business context
3. WHEN a Growth_Play outcome is determined, THE Memory_System SHALL update the Action_History record with success or failure status
4. THE Memory_System SHALL index Action_History records in the RAG_Index

### Requirement 4: Store Technical Maps

**User Story:** As a developer, I want signal definitions and integration schemas indexed, so that the agent interprets platform data correctly.

#### Acceptance Criteria

1. WHEN a Technical_Map is created, THE Memory_System SHALL store it in the S3_Bucket
2. THE Memory_System SHALL support signal definition documents and integration schema documents
3. THE Memory_System SHALL index Technical_Maps in the RAG_Index
4. WHEN an integration schema changes, THE Memory_System SHALL update the corresponding Technical_Map

### Requirement 5: Retrieve Relevant Context

**User Story:** As the agent, I want to search memory for relevant context, so that my recommendations are informed by past experience and business knowledge.

#### Acceptance Criteria

1. WHEN the agent needs context, THE Memory_System SHALL perform a vector search against the RAG_Index
2. THE Memory_System SHALL return the top 5 most relevant documents based on semantic similarity
3. THE Memory_System SHALL include document type and timestamp in search results
4. THE Memory_System SHALL complete retrieval within 2 seconds

### Requirement 6: Parse and Print Memory Documents

**User Story:** As a developer, I want to serialize and deserialize memory documents reliably, so that data integrity is maintained.

#### Acceptance Criteria

1. WHEN a memory document is stored, THE Memory_System SHALL serialize it to JSON format
2. WHEN a memory document is retrieved, THE Memory_System SHALL parse it from JSON format
3. THE Memory_System SHALL validate document structure before storage
4. FOR ALL valid memory documents, parsing then serializing then parsing SHALL produce an equivalent document (round-trip property)

### Requirement 7: Prevent Repeated Failures

**User Story:** As a user, I want the agent to avoid suggesting strategies that previously failed, so that I don't waste time on ineffective tactics.

#### Acceptance Criteria

1. WHEN the agent generates a Growth_Play, THE Memory_System SHALL retrieve similar past Growth_Plays from Action_History
2. IF a similar Growth_Play failed within the past 90 days, THEN THE Memory_System SHALL flag it as a repeated failure risk
3. THE Memory_System SHALL provide the failure context to the agent for decision-making
4. THE Memory_System SHALL consider a Growth_Play similar if semantic similarity exceeds 0.85

### Requirement 8: Integrate with Bedrock Knowledge Bases

**User Story:** As a developer, I want to use Amazon Bedrock Knowledge Bases for vector storage, so that the system stays within AWS Free Tier limits.

#### Acceptance Criteria

1. THE Memory_System SHALL use Bedrock_KB for vector indexing and retrieval
2. THE Memory_System SHALL configure the S3_Bucket as the Bedrock_KB data source
3. WHEN documents are added to the S3_Bucket, THE Memory_System SHALL trigger Bedrock_KB synchronization
4. THE Memory_System SHALL use Amazon Nova Lite for embedding generation to minimize costs
